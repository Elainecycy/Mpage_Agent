## 1 特性概述【必需】

**作为**低代码平台的运营搭建人员，
**我想要**用一段文字描述（页面主题、章节内容、卡片要求）并勾选已上传的素材图，就让 AI 自动生成一份完整的页面配置，
**以便于**把搭建一个活动页的时间从手动拖拽组件的 30 分钟以上缩短到 1 分钟以内，并在生成结果上直接进编辑器微调发布。

## 3.1 特性全景描述【必需】

本次特性建设目标是为低代码平台新增「AI 页面生成」能力，**但本轮迭代仅聚焦纯文字描述驱动的首次生成链路**（描述 + 素材清单 → 页面 JSON → 校验 → 返回平台配置），不涉及示意图视觉识别（第 2 期）、自然语言多轮修改与版本回滚（第 3 期）、交互类组件（按钮/热区）、编辑器改动回灌 AI（双向同步）等后续能力。

特性功能地图（角色 → 活动 → 功能）：

```
角色                活动                        本期功能（✅本期 / ⬜后续 / ⬛非本服务职责）
──────────────────────────────────────────────────────────────────────────
运营搭建人员    上传素材 + 描述需求     ⬛ 平台前端收集，经平台后端转换（含图片上传）后下发
              │
              ├─ （素材已成 URL）       ✅ F1 素材清单接入（接收 manifest，不处理上传/OSS）
              │                       ✅ F2 Prompt 组装（Schema + 示例注入）
              │                       ✅ F3 页面 JSON 生成（LLM 调用 + 自愈重试）
              │                       ✅ F4 两层校验（格式 + 引用完整性 + URL 白名单）
              │                       ✅ F5 生成接口（串联 F1~F4，落库返回）
              │
              ├─ 上传示意图生成        ⬜ 第 2 期（Vision）
              └─ 对话修改页面          ⬜ 第 3 期（ops 协议）

平台后端       转换请求、调 Agent、     ⬛ 上游调用方：完成图片上传产出 URL，调用 F5，
（请求转换层）   回传结果给前端              拿到 pageJson/platformConfig 后回传前端

LLM 网关       提供模型推理            ✅ 默认模型 qwen-plus（冒烟验证 15/15 通过，
                                         见 smoke_test/results.csv），可配置切换
```

技术架构边界（详见设计方案 §3、§4）：

**调用链与上游关系**——Agent 是现有低代码平台的**下游服务**，不直接对接浏览器：

```
平台前端 ──► 平台后端（请求转换层）──► Agent 服务
  上传图片      在此完成图片上传/存 OSS，          收到的就是
  + 描述        把图片处理成 URL，组装 assetManifest    assetManifest（URL 数组）+ 描述
```

- **图片上传与存储不属于本服务**。素材的上传、存 OSS、生成可访问 URL 都发生在平台后端的请求转换环节；请求流转到 Agent 时，图片早已是一组 URL（assetManifest），Agent 不接触图片二进制、不调 OSS、不做文件流处理。
- 因此 Agent 的所有接口入参中，图片一律是 URL；本服务也不向外暴露任何上传类接口。
- 模型只输出**页面 JSON**（平铺组件表 + 文案/图片数据区），不感知平台私有配置格式；JSON → 平台配置的翻译由前端 Mapper 确定性完成。
- 本期 Text / Icon 组件**不携带坐标**，布局由前端预设规则承担。

## 4 功能说明【必需】

### 4.1 素材清单接入（F1）

#### 4.1.1 功能描述

- **图片上传与 OSS 存储不在本服务范围**：素材由平台现有上传能力处理，产出可访问的 URL。Agent 直接接收已经成型的素材清单 `assetManifest`（一组 URL + 元数据），不接触图片二进制、不调 OSS。
- `assetManifest` 为数组，每项含：`url`（OSS/内网 CDN 地址）、`name`（原始文件名，可选）、`note`（用户标注的用途，可选）。manifest 作为生成接口（4.5）的入参随请求传入。
- Agent 侧对 manifest 的职责：
  - 校验 manifest 基本合法（数组非空、每项含 `url`、URL 格式合法），不合法时直接拒绝生成请求并返回明确错误，不进入模型调用；
  - 将 manifest 注入 Prompt（4.2）供模型选图；
  - 以 manifest 的 URL 集合作为校验（4.4）的白名单，拦截模型编造的图片地址。
- 页面 JSON 的 `data.images` 只保留**实际用到**的图片；未用到的素材不进页面 JSON（manifest 由平台/会话侧保留，与本服务无关）。

#### 4.1.2 验收标准

1. 传入合法 manifest（含 1~20 项）时，URL 集合被正确解析为白名单并注入 Prompt。
2. 传入非法 manifest（空数组、缺 url 字段、url 格式非法）时，在调用模型前即返回结构化错误，不产生模型调用开销。
3. manifest 中存在模型未使用的素材时，不影响生成；最终 `data.images` 不包含未引用项。
4. 本服务代码中无任何图片上传 / OSS / 文件流处理逻辑（职责边界自检）。

### 4.2 Prompt 组装（F2）

#### 4.2.1 功能描述

- 实现 Prompt 组装器，按设计方案 §4.3 的固定结构产出 system prompt：角色说明 → 工作步骤 → UI 规则 → 组件 Catalog 字段说明 → 完整 few-shot 示例。
- few-shot 使用设计方案附录 A 的**无坐标版**示例（已在 `smoke_test/system_prompt.md` 验证可用，可直接以其为基线迁移为代码内模板）。
- UI 规则中必须包含：禁止编造图片 URL（只能取自 assetManifest）、禁止为 Text/Icon 输出 position/size、文案图片字段一律 path 引用。
- user message 模板：页面需求描述 + assetManifest JSON。

#### 4.2.2 验收标准

1. 组装产物与设计方案 §4.3 五段结构一致，附录 A 示例作为 few-shot 注入后能通过 4.4 校验器（示例自身合法）。
2. Prompt 模板与 Catalog 定义分离：新增一种卡片类型时，只改 Catalog 字段表与示例，组装代码零改动。
3. 每次生成请求的完整 prompt 可在日志中按 traceId 还原（脱敏后落盘）。

### 4.3 页面 JSON 生成（F3）

#### 4.3.1 功能描述

- 实现 `generate_page_json` 核心流程：组装 Prompt → 经统一 LLM Gateway 调用模型（默认 qwen-plus，温度 0.2，开启 JSON mode 如可用）→ 解析输出 → 交 4.4 校验。
- **自愈重试**：校验失败时，把具体错误列表回灌给模型要求修正，最多重试 1 次（共 2 次调用）；仍失败则返回失败响应，不返回半成品。
- LLM Gateway 为薄抽象层（`complete(prompt, json_schema?)`），模型名、地址、密钥走配置，支持不改代码切换模型。
- 全量调用日志：prompt、原始输出、校验错误、重试次数、延迟、token 用量，按 traceId 关联。

#### 4.3.2 验收标准

1. 跑迭代计划 1.11 的 Eval 集（20~30 条用例）：**Schema 合法率（含 1 次重试后）≥ 98%，卡片类型选择正确率 > 90%，素材命中率 ≥ 80%**。
2. 模型切换仅改配置即生效（用冒烟包 5 用例对切换后模型回归验证）。
3. 单次生成端到端延迟 P95 ≤ 60s（参考冒烟实测均值 15.6s）；超时有明确错误响应。
4. 重试耗尽后的失败响应包含可读的失败原因，不向前端返回未通过校验的 JSON。

### 4.4 两层校验（F4）

#### 4.4.1 功能描述

按设计方案 §4.5 实现，作为生成链路的强制关卡（不做成模型工具）：

- **第一层 JSON Schema 校验**：字段类型、必填、枚举。注意宽严分级——不使用 `additionalProperties: false` 一票否决，多余无害字段由 Mapper 静默丢弃。
- **第二层引用完整性校验** `check_integrity(page, assetManifest)`：
  - id 唯一；有且仅有一个 `root`（component=Page）；
  - children 引用闭合（无悬空 id）、叶子组件不得有 children；
  - 所有 path 引用可在 `data.texts` / `data.images` 解析，文案/图片字段禁止内联裸字符串；
  - `data.images` 每项含 url 且 **url 必须在 assetManifest 白名单内**（防编造，硬失败）；
  - 孤儿组件报错；孤儿 data key 降级为警告并由服务端自动清理（不触发重试）。
- 校验错误信息面向模型可读（指明组件 id 与具体问题），用于 4.3 的回灌自愈。

#### 4.4.2 验收标准

1. 设计方案 §4.5 列出的每条规则均有单元测试，且每条配至少 1 个反例用例（参考 `smoke_test/smoke_check.py` 已验证的正反例集）。
2. 编造 URL 的输出 100% 被拦截，0 漏判（冒烟 case4 类场景）。
3. 孤儿 data key 不触发重试，清理后日志留痕。
4. 校验耗时对单次请求增量 < 100ms。

### 4.5 生成接口（F5）

#### 4.5.1 功能描述

- 新增接口 `POST /api/pages/{pageId}/generate`，Body：`{ assetManifest[], userPrompt }`（assetManifest 即平台上传能力产出的 URL 数组；本期无 `mockupUrl`）。
- 串联流程：校验 manifest（4.1）→ 4.2 组装 → 4.3 生成 → 4.4 校验 → 调用前端 Mapper 同构逻辑生成平台配置（或返回页面 JSON 由前端转换，以前端技改文档定稿的集成方式为准）→ 页面 JSON 与平台配置落库 → 返回 `{ pageJson, platformConfig, previewUrl }`。
- 幂等与并发：同一 pageId 的生成请求串行化，进行中重复提交返回 409。
- 生成结果落库即建立首个版本记录（为第 3 期 revisionHistory 预留表结构，本期只写不读）。

#### 4.5.2 验收标准

1. 迭代计划验收用例全链路通过：输入"做一个新春活动页，头图用 bg.png，主标题新春特惠，下面一个章节放有背景图的文字卡片，再一个章节放图标卡片和规则文案"→ 返回可渲染的 platformConfig。
2. 失败场景（模型超时、校验耗尽、素材缺失）均返回结构化错误码，前端可区分提示。
3. 并发重复提交被正确拦截；生成记录落库后可按 pageId 查询最近一次的 pageJson 原文。
