# 技改文档：AI 生成页面的前端渲染与编辑器集成（第 1 期 · 文字生成页面）

> 上游文档：[设计方案](./ai-page-builder-agent.md) · [迭代计划](./development-iteration-plan.md)（第 1 期，06-18 ~ 07-08）
> 本文档范围：**前端侧**（页面 JSON → 平台基础组件的转换器、布局引擎、编辑器导入、脱离 AI 标记）。
> Agent 服务端见姊妹文档 [tech-spec-phase1-agent.md](./tech-spec-phase1-agent.md)。

---

## 1 特性概述【必需】

**作为**低代码平台的运营搭建人员，
**我想要** AI 生成的页面直接以平台现有基础组件（文字、背景图、图标组件）的形态出现在编辑器里，每个组件都能像手动搭建的一样选中、拖拽、改属性，
**以便于**在 AI 生成的八九不离十的底稿上快速微调细节并发布，而不是面对一个改不动的整体渲染结果。

---

## 3.1 特性全景描述【必需】

本次特性建设目标是打通「AI 输出的页面 JSON」到「平台编辑器可编辑组件」的前端链路，**但本轮迭代仅聚焦单向转换与默认布局**（页面 JSON → 基础组件配置 → 编辑器导入），不涉及编辑器改动逆向写回页面 JSON（双向同步，后续按需求评估）、Text/Icon 按坐标精确定位（第 2 期 Vision 提供坐标后启用）、卡片携带坐标的高保真还原、对话式修改的 diff 预览（第 3 期）等后续能力。

特性功能地图（角色 → 活动 → 功能）：

```
角色                活动                          本期功能（✅本期 / ⬜后续）
──────────────────────────────────────────────────────────────────────
运营搭建人员    触发 AI 生成并预览        ✅ F1 转换器：页面 JSON → 基础组件配置
              │                         ✅ F2 布局引擎：默认排版（无坐标模式）
              │
              ├─ 进编辑器微调            ✅ F3 编辑器导入：生成结果可逐组件编辑
              │                         ✅ F4 "手动编辑后脱离 AI"标记
              │
              ├─ 微调后继续对话改        ⬜ 第 3 期（本期手动编辑后 AI 入口置灰）
              └─ 按示意图精确还原        ⬜ 第 2 期（坐标模式，F2 已预留双模式入口）

Agent 服务     提供页面 JSON             ✅ 契约见设计方案 §3.4（本期组件不带坐标）

编辑器/渲染    渲染与发布                ✅ 复用现有能力，本期不改造编辑器内核
```

核心设计约束（详见设计方案 §3.3 重要区分）：页面 JSON 里的 `Page` / `Header` / `Chapter` / 各类卡片是给模型用的**语义组件**，不是前端组件——转换器把它们**展开为平台现有基础组件的组合**（如 `IconCard` → 图标组件 + 文字组件），容器本身不产出组件、只提供排版上下文。转换必须是纯函数、确定性、可单测。

## 4 功能说明【必需】

### 4.1 转换器：页面 JSON → 平台基础组件配置（F1）

#### 4.1.1 功能描述

- 实现 `mapPageJsonToPlatform(pageJson): PlatformComponent[]`（TypeScript 纯函数库，无副作用、无网络请求）。
- 流程：平铺 `components[]` 按 id 建 Map → 从 `root` 递归还原树 → 逐节点映射：
  - `resolveText` / `resolveImage`：把 `{ "path": "/texts/k" }` / `{ "path": "/images/k" }` 解析为真实文案与图片 URL；
  - `BackgroundImage` → 背景图组件 ×1；`Text` → 文字组件 ×1；`Icon` → 图标组件 ×1；
  - `TextCard` → 文字组件 ×1（套卡片默认样式）；`IconCard` → 图标组件 + 文字组件 ×1~2；`BackgroundTextCard` → 背景图组件 + 文字组件；
  - `Page` / `Header` / `Chapter` 不产出组件，向子节点传递排版上下文（区域纵向偏移、可用宽度）。
- `style` 键（fontSize/color/fontWeight/textAlign/lineHeight）映射到平台文字组件对应属性；模型多输出的未知字段静默丢弃（与 Agent 侧校验宽严分级一致）。
- 对输入做防御性处理：path 解析失败、未知组件类型等异常不抛错中断，按「跳过该组件 + 收集告警」处理并上报（正常情况下 Agent 侧校验已拦截，此处是最后防线）。

#### 4.1.2 验收标准

1. 设计方案附录 A 示例转换结果正确：组件种类、数量、内容与对照表一致（14 个语义组件 → 预期的基础组件列表）。
2. 单测覆盖全部 9 种语义组件与 6 类卡片拆解组合，含异常输入（path 悬空、未知类型）的防御分支。
3. 同一输入多次转换输出完全一致（确定性）；转换 50 组件规模页面耗时 < 50ms。
4. 转换产物中每个基础组件可在编辑器中独立选中、修改、删除，与手动搭建的组件无行为差异。

### 4.2 布局引擎：无坐标默认排版（F2）

#### 4.2.1 功能描述

本期页面 JSON 不含 Text/Icon 坐标，由布局引擎按设计方案 §3.5 规则计算所有组件的位置尺寸（设计稿宽 375px）：

- **头图区域**：高度由头图背景图 `height` 决定（未填时按图片原始比例换算）；主标题垂直居中、水平按 `textAlign`（默认居中），副标题位于主标题下方 8px，Icon 默认贴角（第 1 个左上、第 2 个右上，依次类推）。
- **章节区域**：高度由内容撑开——章节标题（如有）在顶部，其下卡片按 `children` 顺序纵向流式堆叠（左右边距 16px、卡片间距 12px），章节总高度 = 排版结果 + 上下边距；章节背景图拉伸/裁切至该高度，背景图自带 `height` 不生效。
- **卡片内部**：TextCard 高度按文字行数；IconCard 图标左、标题/说明右，按内容撑高；BackgroundTextCard 按背景图比例定高、文字居中叠加。
- 页面总高度 = 头图 + 各章节依次堆叠。
- **为第 2 期预留双模式**：布局入口判断组件是否携带 `position`/`size`，有则直接采用（本期此分支仅留接口与单测桩，不实装视觉验证）。

#### 4.2.2 验收标准

1. 组合用例全覆盖且渲染**不重叠、不溢出、不留异常空白**：三种卡片混排、章节有/无标题、章节有/无背景图、头图含 0~3 个 Icon、单卡片/多卡片章节。
2. 同一章节内卡片顺序与 `children` 数组顺序一致；章节顺序与 `Page.children` 一致。
3. 文字超长场景（标题 30 字、卡片文案 500 字）自动换行撑高，不截断不溢出。
4. 布局结果以显式坐标写入平台组件配置（编辑器中可见可改），而非运行时动态计算——保证用户微调后所见即所得。

### 4.3 编辑器导入与预览（F3）

#### 4.3.1 功能描述

- 生成接口返回后，前端将 platformConfig 经 `editor.load()` 导入现有编辑器，并提供生成结果预览页（复用现有 HTML 渲染链路）。
- 导入的页面与手动新建页面在编辑器中能力完全一致：组件树面板可见全部基础组件、属性面板可改、可保存可发布。
- 生成过程提供加载态与失败态 UI：失败时按 Agent 接口的结构化错误码区分提示（模型超时 / 校验未通过 / 素材缺失），并提供「重新生成」入口。

#### 4.3.2 验收标准

1. 迭代计划验收用例（"新春活动页"一句话描述）端到端走通：生成 → 编辑器导入 → 修改一处文案与一张图 → 保存 → 预览 → 发布。
2. 导入后编辑器无控制台报错；组件树结构与 4.1 转换产物一一对应。
3. 三类失败场景的提示文案与重试入口可用；生成中重复点击被防抖拦截。

### 4.4 "手动编辑后脱离 AI"标记（F4）

#### 4.4.1 功能描述

按设计方案 §4.7 方案 B 实现本期的同步策略：

- 页面记录新增状态字段（如 `aiEditable: boolean`）：AI 生成后为 true；用户在编辑器内**首次保存手动改动**时置为 false 并落库。
- 状态为 false 的页面：AI 相关入口（本期为「重新生成」，第 3 期扩展到对话修改）置灰，悬浮提示文案明确告知"页面已手动编辑，将以编辑器内容为准；如需重新使用 AI 请基于新生成结果"。
- 用户主动选择「重新生成」（对仍为 true 的页面）会整体覆盖当前配置，操作前需二次确认。

#### 4.4.2 验收标准

1. 手动编辑保存后状态正确翻转且持久化，刷新页面、换端登录后状态一致。
2. 置灰入口与提示文案按交互稿验收；不存在"已手动编辑的页面被 AI 静默覆盖"的路径（含并发场景：编辑器保存与生成请求竞争时，以版本检查拒绝后到的覆盖）。
3. 重新生成的二次确认弹窗可取消；确认后旧配置被覆盖且有操作日志。
